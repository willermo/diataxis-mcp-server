# Self-audit of this documentation set

This page explains the sample outputs generated by dogfooding the current MVP against selected repository documentation.

The outputs are advisory. They were generated from deterministic MVP heuristics, not from human review, external APIs, crawling, persistence, RAG, or LLM calls. They should be treated as signals for maintainers to inspect, not as automatic refactoring instructions.

The tree audit sample uses `scope: "sample"`. Its coverage notes should not be read as claims about the full documentation set.

## Inputs

The sample audit covered:

- `README.md`
- `CONTRIBUTING.md`
- `SECURITY.md`
- `LICENSING.md`
- `NOTICE.md`
- `examples/codex/README.md`
- `examples/antigravity/README.md`
- `references/diataxis/diataxis-operational-summary.md`

The generated outputs are checked in as:

- `examples/sample-output/audit-tree.json`
- `examples/sample-output/refactor-plan.json`

## What the MVP reported

The tree audit classified eight pages:

- tutorial: 2 pages
- how-to: 1 page
- reference: 5 pages
- explanation: 0 pages

Because the audit was scoped as a sample, it reported that the sample does not include `explanation` pages and is dominated by `reference` pages. It also reported three pages with mixed-mode signals.

The refactor plan produced advisory `review` and `extract` actions. These actions include suggested target modes and target paths, but the server does not write files or move content.

## How to interpret the result

The output is useful mainly as a first pass over documentation architecture. It highlights places where the current heuristics see mixed user needs, such as README material that combines configuration, conceptual context, and reference-style lists.

The result is not a claim that the current documentation is wrong. Repository-level files often serve more than one practical purpose, and the MVP intentionally uses simple signals. A maintainer should decide whether a recommendation is worth applying.

## Follow-up questions

Useful human review questions include:

- Should the README remain a mixed overview, or should some conceptual material move into explanation docs?
- Should contributor setup instructions become a focused how-to guide?
- Should security, licensing, and notice files stay as repository reference material?
- Should the local Diataxis operational summary remain under `references/`, or should a shorter public-facing explanation page link to it?
